# 儿童学习平台 - 全局功能交互规范
## 一、规范概述
本规范为全项目所有功能模块必须遵循的统一交互标准，目的是保持产品体验一致性，降低开发沟通成本，所有新功能开发、现有功能迭代都必须严格遵循本规范，涉及全局交互更新时必须先同步更新本规范文档，再进行代码开发。
## 二、通用交互基础规范
### 2.1 按钮交互规范
| 状态 | 交互要求 |
|------|----------|
| 正常态 | 遵循对应模块配色规范，文字清晰 |
| Hover态 | 颜色加深/轻微放大，鼠标变为手型 |
| 点击态 | 按压效果（缩小2%或者阴影变化） |
| 禁用态 | 透明度50%，鼠标变为禁止符号，点击无响应 |
- 所有可点击元素的最小点击区域：≥48x48px，适合儿童操作
### 2.2 反馈规范
- **成功反馈**：操作成功后显示绿色提示+简单动画（比如星星、对号弹跳），提示停留2秒自动消失
- **错误反馈**：操作失败显示红色友好提示，不暴露技术错误信息，明确告知用户解决方案
- **加载反馈**：所有异步操作（音频播放、数据请求等）必须显示加载状态，禁止页面无响应让用户等待
### 2.3 操作流程规范
- 儿童使用的功能模块操作步骤不能超过3步
- 所有不可逆操作（比如清空学习记录）必须有二次确认弹窗
## 三、音频播放统一规范（全项目通用）
所有语音/音频播放功能必须遵循以下标准：
### 3.1 状态流转规范
| 当前状态 | 用户操作 | 响应逻辑 |
|----------|----------|----------|
| 未播放/已完成 | 点击播放 | 从头开始播放，按钮变为"暂停"，显示播放状态 |
| 播放中 | 点击暂停 | 语音暂停，高亮同步冻结，按钮变为"继续" |
| 暂停中 | 点击继续 | 从暂停位置无缝继续播放，高亮同步跟进，按钮变回"暂停" |
| 播放中/暂停中 | 切换内容/退出页面 | 自动停止当前播放，重置所有状态 |
| 播放完成 | 自动重置 | 所有状态恢复到初始未播放状态，按钮变回"播放"，可再次点击播放 |
### 3.2 语音合成标准参数
- 优先选择自然中文女声（微软晓晓、阿里云小爱等系统级高质量语音）
- 默认语速：0.60倍（小学教师带读语速，每个字发音清晰，适合儿童跟读）
- 默认音调：1.1（柔和不尖锐，贴近真实教师发音）
- 音量默认1.2倍，可根据场景调整
### 3.3 逐字同步高亮规范
- 高亮颜色采用对应模块主题色，放大1.1倍突出显示
- 智能匹配标点停顿：普通汉字停留400ms，逗号/顿号停留220ms，句号/感叹号/问号停留320ms
- 可搭配指读动画（比如卡通小狗、手指图标）在当前文字上弹跳，增强趣味性
## 四、页面布局统一规范
### 4.1 全局布局规则
- 所有页面采用全屏高度布局（100vh），禁止全局横向/纵向滚动条
- 页面内容区域超出时使用局部滚动，仅滚动对应内容模块
- 整体采用Flex弹性布局，头部导航固定，内容区域自适应高度
- 全局统一边距标准：页面左右内边距固定为18px，模块之间间距固定为18px
- 移除所有页面最大宽度限制（max-width），内容区域占满整个屏幕宽度，不设置固定容器宽度
### 4.2 滚动规范
- 列表类内容（如诗词列表、习题列表）采用独立滚动区域，超出高度自动出现滚动条
- 长文本内容（如诗词正文、阅读理解文章）内部滚动，不影响其他模块位置
- 多栏布局时各栏独立滚动，横向宽度比例固定，滚动不会破坏整体布局
- 滚动条样式统一为细圆角样式，符合儿童产品活泼风格

### 4.4 分栏布局规范
- 古诗词等双栏页面采用3:7固定比例布局，左侧列表栏占30%宽度，右侧内容栏占70%宽度
- 各栏高度自适应填充父容器，不会出现内容溢出导致布局错乱
- 栏与栏之间固定间隔为18px，符合全局间距标准
### 4.3 操作按钮规范
- 核心操作按钮（播放、提交、打卡等）采用固定悬浮设计，吸附在页面/模块底部
- 按钮背景使用半透明白色+模糊效果，不遮挡内容且清晰可见
- 按钮位置不随内容滚动而移动，始终保持在用户可操作区域
## 五、导航跳转统一规范
### 5.1 面包屑导航规范
- 所有二级及以上页面必须显示面包屑导航
- 格式统一：`首页 > 父级页面 > 当前页面`
- 交互规则：首页、父级页面可点击跳转，当前页面高亮显示对应模块主题色，不可点击
- 字体大小16px，灰色显示，hover高亮为全局主色#3399FF
### 5.2 返回操作规范
- 页面左上角不单独设置返回按钮，通过面包屑实现返回逻辑
- 移动端适配可额外增加左上角返回按钮，交互逻辑和面包屑一致
### 5.3 页面切换规范
- 页面切换使用淡入淡出过渡动画，时长0.3秒，避免生硬跳转
- 离开有未保存内容的页面时，弹出二次确认提示
## 五、儿童友好交互专项规范
1. **操作简化**：所有儿童使用的功能避免复杂交互，尽量点击1-2次就能进入学习
2. **反馈生动**：正确操作有正向反馈（鼓励音效、星星奖励、动画效果），错误操作有温柔提示，避免打击学习积极性
3. **防误触设计**：按钮之间保留足够间距，避免误点；重要操作加确认步骤
4. **时长控制**：连续学习超过20分钟弹出休息提示，保护视力
## 六、开发流程规范
### 6.1 开发服务器启动规范
- **规范原则**：开发服务器启动为全局操作，完全由用户手动管理，禁止任何自动启动行为
- **操作流程**：
  1. 代码调整完成后，仅需提交代码修改即可，无需启动服务验证
  2. 如需预览效果，由用户手动执行 `npm run dev` 启动开发服务器并在浏览器中查看
- **严格禁止行为**：
  1. 代码调整后自动执行npm run dev启动开发服务
  2. 自动调用OpenPreview工具打开预览页面
  3. 任何未经用户明确要求的服务启动、预览操作
- **例外情况**：仅当用户明确提出需要验证功能效果时，才可启动开发服务器

### 6.2 代码提交规范
- **规范原则**：仅在功能开发完整、确认无误准备正式提交代码时，才同步生成对应变更记录
- **强制要求**：
  1. 功能开发完成并验证无误后，提交代码前必须同步生成对应变更摘要文档，详细记录完整修改内容
  2. 变更摘要文件命名统一采用格式：`YYYYMMDD-HHMM_变更内容概述.md`
  3. 变更摘要内容必须包含：变更基本信息、详细变更内容、影响范围、验证要点四个部分
  4. 涉及全局规范调整的变更，必须先更新全局规范文档再提交代码
- **禁止行为**：
  1. 完整功能提交时只提交代码不更新对应变更摘要
  2. 变更摘要内容简略、模糊，无法明确具体修改内容
  3. 跳过文档更新直接提交功能代码
  4. 开发过程中未完成的临时修改强制要求生成变更摘要和提交代码

## 七、文档更新规则
1. 本规范为全项目最高优先级交互标准，所有功能开发必须先遵循本规范
2. 涉及全局交互逻辑更新时，必须先更新本规范文档，再进行代码开发
3. 各模块专属交互规范，在对应模块需求文档中单独说明，但不得与本全局规范冲突
4. 规范迭代需要记录版本号和更新日期，历史版本可追溯
